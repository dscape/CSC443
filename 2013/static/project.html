        Project

        CSC443

        1. Introduction

        In the final project, we encourage you to further explore the design principles of physical database systems, and apply these database principles to specific (and interesting) problems.

        2. Possible topics for the project

        You are asked to choose one of the following topics. Aside from the very general description of the topics, you are completely free to make any design, implementation and experimentation decisions.

        • Disk based hash index

        Use disk based hash index to speed up a non-trivial SQL query. You should cover the extensible and linear hashing. Describe how hash index can be use to speed up specific queries of interest. Compare the speedup with SQLite.

        You should identify a broader class of queries that can benefit from your approach.

        • Parallel query processing

        Use parallel programming techniques to speed up a non-trivial query. Compare the speedup with SQLite.

        You should identify a broader family of queries that can benefit from parallelism.

        NOTE: you should implement your project using a true multi-threading environment. This excludes Python and Ruby as they use global shared locking and nonblocking I/O to simulate threading.

        References:

        1. Thread support in C++11
        2. A tutorial on pthread library for C
        3. The Go language and concurrency
        • Parallel sorting

        Explore parallel programming to improve disk based sorting. Evaluate the speedup of sorting with respect to increasing concurrency (say number of threads or processors or machines).

        • Text search with fuzzy string matching

        Build on PL6 to improve text search using keywords.
        You are to work with a large text corpus, for example, the IMDB datasets (available here as plain text files).

        You should explore the usage of n-gram based approximate string matching. You can index the documents by tokening the content into n-grams, and perform search based on matching of the query keywords at the n-gram level. Xapian (recommended) can be used for indexing documents whose terms are n-grams and values as the original text.

        An interesting problem created by approximate string matching using n-gram is that it's not immediately clear on how to highlight the relevant words in the search result. In your project, you should address this issue.

        • Classical algorithms on Big Data

        Implement a classical algorithm (such as, but not limited to: minimal spanning tree, shortest path, k-mean clustering, logistic/linear/auto regression) for data too large to fit into memory. Highlight the application of database technology (sqlite / leveldb / your own implementation) in your design and implementation of the final system.

        As we live in the days of Big Data, you are to collect some data sets of non-trivial size, and design and realize a database architecture to execute a classical algorithm on that data set. Apply the database technologies we covered in this course appropriately. Try to draw some interesting conclusions from the result of the algorithm. Also try to visually present the conclusion.

        Here are some examples. You are free to come up with alternative data sets and analysis.

        1

        You may consider downloading the stack overflow data dump, which contains sufficient data to support trends in the popularities of programming languages, API usages and programming practices.

        2

        DBLP, the scientific publication, data set can be downloaded here.

        It contains the (very large) social network describing the collaborative relations among researchers.

        3

        IMDB, the movie data set, contains very rich descriptions on movies, producers, actors, box office performances, movie genre, user keywords and ratings.

        It would be interesting to see the trends in movie making and audiences preferences over time.

        You will almost certainly run into performance issues. You can preprocess the data set by first reducing its size so it becomes tractable by your implementation, while still large enough for interesting results.
